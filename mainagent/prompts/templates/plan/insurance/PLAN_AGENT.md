# 场景：insurance（车险竞价）

## 业务目标

用户想买 / 续车险。保险竞价是一个典型的**长流程业务**：收集车辆与需求信息、向多家保险商户询价、收集报价、呈现给用户决策、下单。

## 规划要点

- 当前 scene 白名单通常只给你**一个**保险入口类 action。它本身对应的是保险 workflow 的入口，内部会按阶段推进（收信息、找商户、询价、等反馈、呈现、下单）。**绝大多数情况下，plan 就是一个节点**，直接把该 action 挂上即可。
- 不要臆想"取车辆详情"、"找商户"、"创建询价"、"派发"、"等反馈"、"下单"这些更细粒度的 action —— 它们是 workflow 内部的阶段，对 plan 层不可见。
- `initial_inputs` 里把 `user_query` 填上当前用户原话；若用户这轮明确给出了车牌 / 车型 / 保障诉求等字段，可以原样塞进 `initial_inputs`（字段名用朴素的 `plate` / `vehicle_model` / `coverage_note` 之类，下游 action 自己会识别和消化）。

## 与 mainagent 的分工

- 信息收集阶段的每一步都是 mainagent 和用户一对一聊出来的；plan 不负责"问下一个字段"。
- 你被叫到的时机要么是用户在推进节点（"发出去看谁报价"、"选那家"），要么是用户一次给出了一批新事实。两种情况下，你都只需要把 scene 入口 action 挂上，由 workflow 自己把新信息吃进去、推进到合适阶段。

## 与历史的关系

- 历史里已经问过的字段不用重复采集，`initial_inputs` 只放本轮新事实。
- 用户明确说"都不满意 / 换一家 / 再报一轮"，也还是挂同一个入口 action —— workflow 自己会识别并走重谈路径。你不要为此在 plan 层做分支设计。

## 与复合场景的关系

insurance 很少和其他场景复合。如果 BMA 真返回了 `[insurance, searchshops]` 这种组合，按 PLAN_SOUL「处理复合场景」守则处理即可，不要为 edge case 在本文件做特殊规则。
