# 场景：insurance（车险竞价）

## 业务目标

用户想买 / 续车险。系统要帮用户：

1. **收集车辆 + 需求信息**（车型、车牌、保障诉求、期望返现等）
2. **找保险商户**（本地可服务的保险商户）
3. **向商户发询价，等待报价反馈**
4. **把多家商户的方案整理后呈现给用户，等用户决策**
5. **下单 + 确认订单**

相比 searchshops（一次就能搜出结果）或 searchcoupons（检索类），insurance 是**典型的长流程竞价业务**，一次 Temporal workflow 跨度可能以小时/天计。

## 与 mainagent 的分工（规划前要先想清楚）

- **信息收集阶段**通常不走你（plan）—— 那是 mainagent 和用户对话一步步问出来的（车型、车牌、需求、返现期望等），每一步都是用户回复 → 系统记一笔 session_state。
- 你被叫到的时机一般是 **信息齐了、要推进竞价 / 等反馈 / 呈现方案 / 下单** 这些"后台驱动"的环节；或者 **信息还缺但用户消息里一次给了新事实**，需要记录并立即往下推进。
- 所以你规划出的 plan 往往**只覆盖"本轮用户消息触发的一小段"**，不是从头到尾整条 insurance 链路。

## 规划阶段（按典型推进顺序）

| 阶段 | 用户视角 | 你可能需要排的动作 |
|---|---|---|
| **准备** | 车和需求都给齐了 | 校验 / 取车详情 / 取需求摘要 → 找商户（可能还要查营业时间 / 扩大范围兜底）|
| **竞价** | "帮我发出去看谁报价吧" | 创建询价单 → 派发给商户 → 等商户反馈（异步 signal） |
| **决策** | "我看看哪家"或商户反馈回来了 | 把 N 家方案整理 / 呈现给用户 → 等用户决策（异步 signal） |
| **下单** | "选那家" | 创建订单 → 确认订单 |
| **重谈** | 用户对报价不满 / 到期未决 | 通知商户重新报价（notify_requote 类），然后回到竞价等反馈 |

## 常见规划形态

**准备 + 竞价一步到位**（信息已齐，用户说"那就让他们报吧"）：

```
取车辆详情 ──┐
取需求摘要 ──┼──→ 找商户 ──→ 查营业时间 ──→ 创建询价 ──→ 派发 ──→ 等反馈 ──→ 合并回复
```

**反馈回来了，只呈现**（workflow 被商户 signal 唤醒后，拿到最新反馈）：

```
整理商户反馈 ──→ 呈现给用户 ──→ 合并回复
```

**用户选了谁，下单收尾**：

```
创建订单 ──→ 确认订单 ──→ 合并回复
```

## 规划时要注意的几点

- **这个场景并行机会少，大多数是强串行**。不要强行让看起来无关的节点并行 —— 竞价链上 "派发 → 等反馈 → 呈现" 每一步都必须有前驱。
- **等待式动作的 `depends_on` 要压得最紧**。像"等商户反馈"、"等用户决策"这类会 block 的动作，它后面的节点必须显式 `depends_on` 它，**别让 LLM 误以为可以和它并行**。
- **找商户前置 + 扩商户兜底**：如果白名单里有"扩大商户范围"类的 action，通常只在"本地商户为空"才挂上；初版规划**不要**为防御性而预加。
- **用户画像 / 车辆详情 / 需求摘要**往往是多个下游动作的共同上游 —— **建一个 profile-like 前置节点，多个节点分叉依赖它**，别重复取。
- **DSL 是 DAG，不支持循环**。如果本轮用户消息暗示"要再报一轮"，规划一个通知重新报价 + 再等反馈的**线性延伸**即可，不用写循环结构（真正的多轮由 workflow / orchestrator 跨轮驱动，每轮重新调 /plan）。

## 与复合场景的关系

insurance 很少和 searchshops / searchcoupons 复合（它本身就是"找保险商户 + 谈价"的自足流程）。如果 BMA 真返回了 `[insurance, searchshops]` 这种，多半是用户提到了"顺便看看修车店"之类的无关诉求 —— 按 PLAN_SOUL「处理复合场景」守则处理即可，不要为这个 edge case 在 insurance 内部做特殊设计。

## 与历史的关系

- 历史里已经问过的信息（车牌 / 期望返现等），**不要重复采集**。如果 initial_inputs 里已经透传，就跳过信息类的 action，直接进入下一阶段。
- 历史里如果已经出现过"商户反馈列表"或"呈现过的方案"，本轮规划往往是**下单阶段**（用户在选谁），不是重新竞价。
- 用户明确说"换一家"、"都不满意"时，规划走**重谈路径**（通知重新报价 + 再等反馈），而不是从头找商户（除非白名单暗示要重新找）。
