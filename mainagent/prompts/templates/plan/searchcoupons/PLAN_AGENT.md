# 场景：searchcoupons（搜优惠活动）

## 业务目标

用户想找当前可领 / 可用的汽修相关优惠活动（满减券 / 折扣项目 / 套餐活动等）。你要规划出从「用户粗糙诉求」到「一份按相关度排序的优惠列表（或已交给回复节点）」的完整执行路径。

## 常见规划形态

**纯优惠查询**：

```
按用户画像+地域筛选优惠 ──→ 排序/过滤 ──→ 交给回复节点
```

**与商户组合（复合场景 searchshops + searchcoupons 常见）**：

```
拉画像 ──┬──→ 搜商户 ─┐
         │              ├──→ 合并回复节点
         └──→ 搜优惠 ───┘
```

## 规划时要注意的几点

- **优惠依赖用户画像**：车型、偏好、常去商户这些都会影响优惠匹配度；如果 initial_inputs 里没有画像，就得前置一次 `fetch_user_profile` 类 activity（如果白名单里有）。
- **与商户场景组合时共用上游**：两个场景都要画像 → 只建一个 `profile` 节点，`depends_on` 分叉（见 PLAN_SOUL「处理复合场景」第 2 条）。
- **可用期限 / 库存**：如果白名单里有校验优惠有效性的 activity，一般挂在搜优惠之后、出回复之前。
- **末节点合并**：和其他场景组合时，**不要**单独起一个"只出优惠回复"的末节点；用统一的 `render_reply` 把商户 + 优惠一起合成一份回复。

## 与历史的关系

如果历史里用户已经明确了某家商户（"就上次那家"），搜优惠时可以把 shop_id 透传到 initial_inputs，让搜优惠的 activity 按店筛选，而不是全域搜。
