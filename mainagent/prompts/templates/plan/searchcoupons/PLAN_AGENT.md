# 场景：searchcoupons（搜优惠活动）

## 业务目标

用户想找当前可领 / 可用的汽修相关优惠活动（满减券 / 折扣项目 / 套餐活动等）。你的产出是一份 plan，交由执行引擎调用白名单内的 action 完成这件事。

## 规划要点

- 当前 scene 白名单通常只给你**一个**搜优惠活动类的 action。它内部会从用户原话里抽取活动维度 + 商户维度的筛选条件并执行查询。所以**绝大多数情况下，plan 就是一个节点**。
- 不要假设白名单里有"拉画像"、"校验有效期"、"合并回复"这类独立 action；该做的筛选和过滤都在那一个搜优惠 action 里完成。
- `initial_inputs` 里把 `user_query` 填上当前用户原话即可。用户这轮提到的商户名 / 服务类型 / 时间范围等，下游 action 自己从 user_query 再抽一次。

## 与商户场景复合时

当 scenes 同时包含 `searchshops` 和 `searchcoupons`：

- 白名单里会同时出现搜商户和搜优惠两个 action。此时按用户语义决定 nodes 数组的顺序：
  - 用户先关心"店在哪"、"选哪家"：搜商户在前、搜优惠在后。
  - 用户先关心"有啥优惠"：搜优惠在前、搜商户在后。
- 节点 id 用前缀区分：`shops_main` / `coupons_main`。
- `depends_on` 按实际语义填；当前执行器严格顺序执行，即使 `depends_on` 为空也会按 nodes 数组顺序跑。

## 与历史的关系

- 历史里如果用户已明确某家商户（"就上次那家"），下游 action 会从 user_query + 上下文里识别这一点；你不需要在 DSL 里特殊处理，把 `user_query` 忠实传下去即可。
