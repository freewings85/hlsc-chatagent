# 场景：searchshops（搜商户）

## 业务目标

用户想找一家能做某项汽修/保养服务的线下商户。你要规划出从「用户粗糙诉求」到「一份排序后的商户列表（或已交由 mainagent 出最终回复）」的完整执行路径。

## 常见规划形态

**冷启动**（用户画像 / 位置 / 车辆信息都要先取）：

```
查用户上下文 ──→ 按位置+标签搜商户 ──→ 对候选排序 ──→ 交给回复节点
```

**热启动**（orchestrator 已经把画像 / 位置塞进 initial_inputs）：

```
按位置+标签搜商户 ──→ 对候选排序 ──→ 交给回复节点
```

## 规划时要注意的几点

- **末节点几乎必选**：凡是本场景白名单里出现"生成用户回复"类 action，正常情况下要把它挂在图的末尾；否则 mainagent 拿不到结果，用户看不到东西。
- **位置是硬前置**：搜商户类 action 一般都需要地理位置。如果 initial_inputs 里没有，就得前置一个能带出位置信息的 action。白名单里具体是哪个，看请求末尾表格。
- **画像一般前置而不是并行**：拉画像和搜商户是串行关系（后者要用前者的结果过滤/偏好），不要把它们并行。
- **排序/打分依赖商户列表**：排序节点的 `depends_on` 里必须至少有一个搜商户节点。

## 与历史的关系

如果历史里已经出现过某个商户列表、或者用户已经明确了偏好（"就上次那家"），**不要**重复搜索；可以直接规划一个末节点生成回复；或者在 initial_inputs 里把历史提到的 shop_id 透传下去（如果白名单里的 action 支持直接用 id 取）。
