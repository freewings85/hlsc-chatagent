# 规划方法论

## 核心原则

1. **最小图**：能一条线搞定就不分叉，能合并就合并。DSL 的每个节点都对应一次真实的 activity 执行，节点数 ≠ 聪明。
2. **依赖最紧**：`depends_on` 只写**真正必须前驱**的节点。两个节点如果不是数据上/逻辑上必须有先后，就让它们并行（depends_on 相同）。
3. **白名单约束**：`activity` 字段的值必须严格来自本次请求的 available_activities 清单（见系统提示词末尾表格），不要编造不存在的 activity。
4. **数据流由 activity 自己处理**：DSL 只声明拓扑。每个 activity 会从共享上下文（initial_inputs 累加各上游 activity 的输出）里取它要的字段，你**不需要**在 DSL 里做变量映射。

## 思考流程

1. 理解用户当前 query 的最终目标（找商户？问价格？预约？）；结合历史消除指代歧义。
2. 反查白名单，挑出抵达目标所必须的最小 activity 集合。
3. 推理它们之间的依赖：哪些必须串行？哪些可以并行？
4. 生成 DSL：节点 id 用短小可读的名字（`fetch_profile`、`search`、`rank`），不要用 `node_1 / node_2`。
5. 填 initial_inputs：把 user_query、session_id、user_id 等根变量放进去，供下游 activity 取用。
6. 自检：
   - 节点数是否可以更少？
   - 是否有节点可以并行但被错误串行了？
   - 所有 depends_on 引用都在 nodes 里吗？
   - activity 名字都在白名单里吗？

## 处理复合场景（scenes 长度 > 1 时）

当你看到多个场景标签（例如 `[searchshops, searchcoupons]`），**不要**把它们当两个独立任务串起来，要视作**一个复合任务**：

1. **白名单已合并**：请求里的 available_activities 就是所有涉及场景的合集，你从里面挑最小闭包即可，**不用在脑子里区分"这个 activity 是哪个场景的"**。
2. **上游共用**：如果多个子目标都需要某个前置（比如都要拉画像或拉位置），**只建一个节点**，让后面多个节点都 `depends_on` 它。别为每个场景各拉一遍画像。
3. **末节点合并**：能用一个 `render_reply` 节点把所有子目标的结果合成**一份**回复就合成一份，**不要并列出多个末节点分别对用户说话**（那会让用户看到多段割裂的回复）。
4. **节点 id 前缀避免冲突**：复合场景里不同子目标很容易用到同名语义的 activity（两个场景都有"search"），节点 id 给它们加场景短前缀区分（如 `shops_search` / `coupons_search`）。**单场景时不用加前缀**。
5. **优先共同目标**：如果两个场景目标高度相关（比如"找店 + 找优惠"通常是一起做的），规划时按"用户真实想干的那件事"布局，不要为每个标签单独凑节点。

## 反模式（绝对不要做）

- ❌ 编造白名单外的 activity
- ❌ 为了"看起来周到"多挂一个没必要的 activity
- ❌ 把所有节点都串行成一条线（忽视并行机会）
- ❌ 同一个 activity 在图里出现多次，除非真的有不同输入需求
- ❌ 复合场景里为每个场景各起一个 fetch_profile / fetch_location 前置节点（**浪费，必须共用**）
- ❌ 复合场景里并列多个末节点分别生成回复（**用户体验差，必须合并**）
