把用户当前表达的"找优惠活动条件"静默登记到会话，由系统按单一 query 树拆分成活动 +
商户两维度去联合查询并把结果带回来。

这是一次**内部登记**——调用对用户不可见。不要在回复里预告"我帮你搜一下""保存中"，
也不要把参数 JSON 或清单贴给用户看。用户看到的只是你对他说的话，搜索是你和系统之间的事。

**要么调这个工具，要么直接对用户说话，不要吐出规划文本**（"我需要把 XXX 加进去……"
这种思考过程不是给用户看的）。决定不充分就追问；决定了就调工具。

## 何时调

用户给的找优惠活动条件已经清晰到可以组出一棵 `query` 树（结构见当前步骤 instruction
里的 COUPON_SEARCH_FIELDS 规范）。字段不分"归活动还是归商户"——按字段语义落进
LEAF.params 即可，后端会自己按字段名拆成活动维度和商户维度。补充 / 修正条件时
**整包重新提交**（非增量）。

## 何时不调

- 用户只闲聊 / 表达情绪 / 问非优惠活动相关问题
- query 组不出来（连一个有效字段都没有），先追问，不要塞空壳
- 用户明确是别的意图（预订下单、查商户评分等），跟用户走

## 返回值

登记完会回一段文字（如"已为用户展示 3 条优惠活动"或"未找到匹配的优惠活动"）——
给你判断下一步用的线索，不是给用户看的原话。真实活动详情通过前端事件直接渲染，
你**不要复述活动文案**，按对话语气说一句自然的回应即可。

## 参数

`coupon_search_info`：整包 JSON，包含 orderBy（可选）/ limit（可选）/ query（查询树）。
LEAF.params 的可用字段以 instruction 里的 COUPON_SEARCH_FIELDS 表格为准，不要从这里推测。
