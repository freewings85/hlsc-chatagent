把用户当前表达的"找优惠条件"静默登记到会话，由系统按两个维度（活动本身 +
商户）去做联合查询并把结果带回来。

这是一次**内部登记**——调用对用户不可见。不要在回复里预告"我帮你搜一下""保存中"，
也不要把参数 JSON 或清单贴给用户看。用户看到的只是你对他说的话，搜索是你和系统之间的事。

**要么调这个工具，要么直接对用户说话，不要吐出规划文本**（"我需要把 XXX 加进去……"
这种思考过程不是给用户看的）。决定不充分就追问；决定了就调工具。

## 何时调

- 用户给的找优惠条件已经清晰到至少一个维度可以组出 `query`（结构见当前步骤 instruction 里的
  ACTIVITY_SEARCH_FIELDS / SHOP_SEARCH_FIELDS 规范）
- 两个维度**任一**有 query 就可以调：
  · 用户只说活动特征（"送洗车的优惠"、"满减活动"）→ 只填 `activity_search_info`
  · 用户只说位置/商户特征（"附近哪家店有优惠"）→ 只填 `shop_search_info`
  · 两个都有 → 两个都填，会 AND 联合查询
- 用户补充 / 修正条件时，把对应维度**整包重新提交**（非增量）

## 何时不调

- 用户只闲聊 / 表达情绪 / 问非优惠相关问题
- 两个维度都不成立（都没有能组 query 的条件）→ 先追问，不要塞空壳
- 用户明确是别的意图（预订下单、查商户评分等）→ 跟用户走

## 返回值

登记完会回一段文字（如"已为用户展示 3 条优惠活动"或"未找到匹配的优惠活动"）——
给你判断下一步用的线索，不是给用户看的原话。真实活动详情通过前端事件直接渲染，
你**不要复述活动文案**，按对话语气说一句自然的回应即可。

## 参数

两个参数——都可选，**至少传一个**（两个都空会直接忽略不调后端）。结构见当前步骤
instruction 里两份 schema 的表格，以那里为准，不要从这里推测。
