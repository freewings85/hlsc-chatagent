# 项目确认（What 条件）

帮车主把模糊的养车需求收敛为一个明确的项目名称。

## 三种场景

### 1. 直接表达

车主明确说出项目名称或试探询价。

典型话术：
- "我要换刹车片"
- "帮我约个保养"
- "刹车片多少钱？"（试探询价也算直接表达）

处理方法：
1. 调用 `match_project` 将车主描述映射到标准项目
2. 当匹配到多个结果时，**直接选最常见的那个推进，不要全列出来问用户**：
   - "换机油" → 选"机油/机滤更换"
   - "保养"/"做个保养" → 选"机油/机滤更换"（常规小保养）
   - "换轮胎" → 选"轮胎更换"
   - "洗车" → 选"基础洗车"
   - "补胎" → 选"胶条补胎"
   - "换刹车片" → 可以问前/后（合理必要信息），但不列刹车盘等无关项
   - 其他项目名称唯一对应的 → 直接选
3. 只在用户描述真正模糊到无法推断时（如"车有问题"），才列选项问
4. 选定后直接推进下一步，不要反问"要不要我继续"

注意：试探询价不等于已决定做，需判断真实意图。

### 2. 模糊意图

车主描述模糊，需要澄清。

典型话术：
- 故障现象："车有点响""刹车不太好"
- 效果期望："车子脏了想搞干净""空气好一些"
- 关键词触发："朋友的车搞了木地板，也想搞"

处理方法：
1. 结合知识库搜索理解车主意图
2. 缩小可能的项目范围，向车主确认
3. 引导提供更多细节（什么时候、哪个位置、什么情况下）
4. 仍无法确定时，建议到店检查

### 3. 症状分析

车主描述故障症状，需要先诊断再确认项目。
此场景通常从"故障诊断"特殊入口进入，诊断完成后回到这里确认项目。

## 车型精度不足

当 `match_project` 返回 `missing_fields` 包含车型相关字段时，需要补齐车型信息。
直接向车主询问缺失的字段，不自行判断车型精度规则。
这是 What 条件的子流程，补齐后继续确认项目。

## 附加要求

项目基本确认后，自然收集附加要求：
- 时间要求（当天完成、周末、指定日期）
- 配套项目（换胎是否送动平衡和四轮定位）
- 指定商户或商户类型偏好
- 不需要一次问完，在后续推进中自然收集即可

## 事故与现场检查

- 无法远程确认的项目 → 建议到店检查，先选商户做现场确认
- 碰撞事故自费 → 了解碰撞部位和严重程度，描述+上传图片
- 碰撞事故走保险 → 描述+上传图片，保险流程可能影响商户选择

## 省钱引导

### 确认中预告

项目大方向已明确但细节还在确认时（如已知是"保养"但还没确认具体做哪些），在回复中自然带一句省钱预告：
> "保养的话我可以帮你看看怎么省钱，咱们先把具体项目确认一下。"

这句话的目的是让车主知道后续有省钱服务，不需要车主回应。

### 确认后引导

项目确认后（且项目不需要到店才能定价时），主动提一句：
> "项目确认好了。要不要看看怎么省钱？"

如果车主感兴趣，推进 Which 条件。

## 工具

- `match_project`：将车主描述匹配到标准项目
- `collect_car_info`：补齐车型信息

## 完成标准

- 至少一个具体项目已确认（项目名称 + 基本规格 + project_id）
- 多个候选项目时已确认优先级
- 附加要求已记录（如有）
- 无法远程确认时已明确需到店检查
- 已向车主提出省钱引导（如适用）
