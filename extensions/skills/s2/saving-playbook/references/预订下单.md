# 预订下单

把已确认的项目、商户、省钱方案汇总成预订，完成下单。

## 三种预订模式

### 标准预订（standard）

车主选择某个商户的报价直接预订。

适用场景：项目明确、商户已选、价格已知。

流程：
1. 确认券型（10 元券 or 9 折券），两种都符合时默认 9 折券
2. 确认价格（车主已知目标价格，或给出行情价后车主同意）
3. 确认服务时间（见下方"时间确认"）
4. 调用 `confirm_booking` 将预订信息发给前端确认
5. 判断车主回复并处理（见下方"回复处理"）

### 委托预订（commission）

车主出一口价，委托推送给多家商户。

适用场景：车主有明确的价格预期，希望多家商户响应。

流程：
1. 确认服务时间
2. 确认车主的一口价
3. 调用 `confirm_booking` 发给前端确认
4. 判断车主回复并处理

### 竞标预订（bidding）

车主给出项目要求，由多家商户竞价报价。

适用场景：仅限 9 折项目，车主希望商户竞价。

流程：
1. 确认 9 折券规则和商户范围
2. 确认服务时间
3. 调用 `confirm_booking` 发给前端确认（price 留空，由商户竞价决定）
4. 判断车主回复并处理

## 时间确认

在调用 `confirm_booking` 之前，必须先确认服务时间。

### 紧急程度判断
- **实时服务**（马上到店）：救援类项目（搭电、拖车、换胎等），省时为最高优先级
- **可预订服务**（可等几天）：非紧急项目，可选择合适日期

### 可预订服务注意事项
- 避开旺季高峰期和节假日前几天
- 避开商户休息时间
- "话痨AI预订9折"需提前至少 2 天预订

### 完成条件
满足任一即可：
- 到店时间已确认（具体日期或时间范围）
- 车主明确表示时间灵活，由商户排期

## 回复处理

`confirm_booking` 返回车主原始回复文本，按内容判断意图：

| 车主回复 | 处理方式 |
|---------|---------|
| "确认""没问题""可以" | 告知预订已提交，流程完成 |
| "不用了""取消" | 告知已取消，流程完成 |
| "换一家店" | 回退到阶段 4（商户选择） |
| "改到周末""下周再去" | 更新时间，重新调用 `confirm_booking` |
| "去掉洗车""加上空调清洗" | 回退到阶段 1（项目确认） |
| "能便宜点吗" | standard：价格由商户决定，建议换商户或项目；commission：更新一口价重新确认；bidding：价格由竞价决定 |
| "备注车在地下二层" | 更新备注，重新调用 `confirm_booking` |
| 无关内容 | 引导车主确认或说明想调整什么 |

注意：
- plan_mode 由 agent 根据条件自动判断，车主不直接选择模式
- 车主调整商户/时间/项目后，agent 应重新评估 plan_mode

## 过渡型方案

前置条件不全时的变通方案：

| 场景 | 处理方式 |
|-----|---------|
| 无 VIN | 先订简单项目（如洗车），让商户协助提供车架号 |
| 项目不确定 | 先订洗车 + 全面检查，到店后再确认 |
| 只要联系方式 | 给商户联系方式，车主自行线下对接 |
| 让商户主动联系 | 按需求搜索商户，让商户联系车主 |
| 想先谈价再订 | 话痨代谈判，达到目标后再标准预订 |
| 车主已在现场 | 确认商户是否认证，是则直接走标准流程 |

## 工具

- `confirm_booking`：汇总预订信息发给前端，返回车主回复文本

## 完成标准

- 预订信息已确认提交（或车主取消）
- 结果已告知车主
