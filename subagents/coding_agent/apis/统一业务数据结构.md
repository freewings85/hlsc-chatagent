# 统一业务数据结构草案

这份文档不是某一个具体 API 的说明，而是给 agent-facing API 设计用的统一数据模型。

目标：
- 让 mainagent 和 codingagent 都读得懂
- 让查询、筛选、排序、比较更直接
- 保留真实业务概念，不丢信息
- 避免把后端原始返回结构直接暴露给 agent

## 一、统一命名原则

- `project_id / project_ids`：业务里的项目 id
- `car_model_id`：业务里的车型 id
- `shop_id / shop_ids`：商户 id
- `user_id`：用户 id
- `shop_type_id / shop_type_ids`：商户类型 id
- `primary_part_id / primary_part_ids`：标准词 / 标准配件 id

说明：
- agent-facing API 文档和返回结构统一使用上面的名字
- 后端内部如果暂时仍使用旧字段名，那只是实现细节，不进入 agent 文档层

## 二、总的返回设计原则

所有面向 agent 的查询结果，尽量都遵守下面的思路：

1. 返回结果先表达“这次任务的结果”，而不是回传底层全量树形结构
2. 固定返回“基础业务对象”
3. 再补“本次任务真正依赖的上下文字段”
4. 如果是排序、筛选、比较任务，必须带上排序 / 筛选 / 比较所依据的关键事实

建议统一采用：

```json
{
  "query": {},
  "items": [],
  "summary": {}
}
```

含义：
- `query`：本次查询的关键信息，便于上层理解这次结果是怎么来的
- `items`：结果列表
- `summary`：总数、范围、排序方式、说明等摘要信息

不是每个 API 都必须机械使用这 3 个字段，但整体方向建议一致。

## 三、商户数据结构

### 3.1 商户基础对象 `shop`

```json
{
  "shop_id": 48,
  "shop_name": "某某门店",
  "shop_type_id": 34,
  "shop_type_name": "4S店",
  "address": "上海市嘉定区...",
  "province": "上海市",
  "city": "上海城区",
  "district": "普陀区",
  "latitude": 31.287737,
  "longitude": 121.326912,
  "distance_m": 216,
  "rating": 4.6,
  "trading_count": 120,
  "phone": "17522536027",
  "opening_hours": "09:00-18:00",
  "tags": ["洗车", "保养"],
  "images": []
}
```

说明：
- 这是所有商户相关返回里都应尽量稳定存在的基础信息
- 上层如果只是做普通商户确认，读到这里基本就够了

### 3.2 商户搜索结果 `shop_search_result`

```json
{
  "query": {
    "project_ids": [516],
    "latitude": 31.28,
    "longitude": 121.32,
    "radius_m": 10000,
    "sort_by": ["distance", "rating"]
  },
  "items": [
    {
      "shop": {}
    }
  ],
  "summary": {
    "total": 12
  }
}
```

说明：
- 如果只是标准商户搜索，`items` 里只需要放 `shop`
- 不要把价格、报价方案、配件树混进这个结果里

### 3.3 商户比较结果 `shop_comparison_result`

```json
{
  "query": {
    "goal": "找出洗车最便宜的两家商户"
  },
  "items": [
    {
      "shop": {},
      "comparison_data": {
        "rank": 1,
        "matched_price": 25.0,
        "price_text": "¥25",
        "comparison_reason": "本次比较按基础洗车价格升序排序"
      }
    }
  ],
  "summary": {
    "total": 2
  }
}
```

说明：
- 一旦结果是“按价格/评分/距离/营业时间筛出来的”，就不能只返回商户基础信息
- 必须返回本次比较真正依赖的数据

## 四、项目数据结构

### 4.1 项目基础对象 `project`

```json
{
  "project_id": 516,
  "project_name": "前刹车片更换",
  "project_simple_name": "前片",
  "need_car_model": true,
  "need_vin": false,
  "contain_material": true,
  "description": "",
  "keywords": ["前刹车片", "制动摩擦"],
  "unit": "一对"
}
```

说明：
- 这是项目相关能力里的最小通用对象
- `need_car_model / need_vin` 比旧的字符串枚举更适合 agent 直接理解

### 4.2 项目搜索结果 `project_search_result`

```json
{
  "query": {
    "search_text": "更换刹车片"
  },
  "items": [
    {
      "project": {},
      "match_data": {
        "similarity": 0.96
      }
    }
  ],
  "summary": {
    "total": 5
  }
}
```

说明：
- 如果是从用户自然语言里检索项目，就把相似度、触发来源这类信息放到 `match_data`
- 不要把项目详情、配件详情、推荐逻辑全混在搜索结果里

### 4.3 项目详情结果 `project_detail_result`

```json
{
  "items": [
    {
      "project": {},
      "detail_data": {
        "first_maintenance_mileage": 0,
        "first_maintenance_time_month": 0,
        "maintenance_mileage": 0,
        "maintenance_time_month": 0,
        "related_projects": [],
        "related_parts": [],
        "shop_type_scope": []
      }
    }
  ]
}
```

说明：
- 详情结果里可以有更多信息，但也仍然要围绕项目对象组织
- 配件、商户类型、保养周期都属于详情上下文，不应该反过来成为顶层主结构

### 4.4 项目关系结果 `project_relation_result`

```json
{
  "query": {
    "project_ids": [516]
  },
  "items": [
    {
      "project_id": 514,
      "project_name": "火花塞更换",
      "relation_type": "related"
    }
  ]
}
```

说明：
- 关系查询要明确关系类型
- 不要只返回一串 id

## 五、报价数据结构

报价是现在最复杂、也最需要重构的一块。

核心原则：
- agent 不应该直接消费原始的“商户 -> 项目 -> 方案 -> 配件组 -> 配件项”全量嵌套树
- 默认应返回“本次任务最相关的报价结果”
- 只有在任务明确需要时，才下钻到配件层

### 5.1 报价基础对象 `quotation`

```json
{
  "project_id": 516,
  "project_name": "前刹车片更换",
  "plan_name": "国货精品",
  "plan_type": "DOMESTIC_QUALITY",
  "total_price": 60.06,
  "price_text": "¥60.06~¥60.06"
}
```

### 5.2 商户报价结果 `shop_quotation_result`

```json
{
  "query": {
    "project_ids": [516],
    "car_model_id": "lavida_2021_15l",
    "shop_ids": [54, 55]
  },
  "items": [
    {
      "shop": {},
      "quotation": {}
    }
  ],
  "summary": {
    "total": 2
  }
}
```

说明：
- 对于“比哪家便宜”这类任务，默认这一层就够了
- 不要一上来把配件树全展开

### 5.3 报价比较结果 `quotation_comparison_result`

```json
{
  "query": {
    "goal": "比较附近 5 公里内前刹车片更换价格"
  },
  "items": [
    {
      "shop": {},
      "quotation": {},
      "comparison_data": {
        "rank": 1,
        "comparison_basis": "按 total_price 升序",
        "price_gap": 20.0
      }
    }
  ],
  "summary": {
    "total": 3,
    "lowest_price": 60.06,
    "highest_price": 120.0
  }
}
```

说明：
- 一旦做比较，必须把比较依据显式返回
- 这样上层 agent 才能稳定解释“为什么是这几家”

### 5.4 配件明细结果 `quotation_part_detail`

```json
{
  "project_id": 516,
  "project_name": "前刹车片更换",
  "plan_name": "国货精品",
  "parts": [
    {
      "primary_part_id": 933,
      "primary_part_name": "前刹车片",
      "items": [
        {
          "part_item_id": 80125525,
          "part_item_name": "前刹车片主推-1K0698151E",
          "price": 30.03,
          "brand_origin": 1,
          "related_oems": ["1K0698151E"]
        }
      ]
    }
  ]
}
```

说明：
- 只有在任务确实需要说明配件、品牌档次、OE 号、配件价格构成时，再返回这一层
- 不要把它作为默认报价结果

### 5.5 行情参考价结果 `market_reference_result`

```json
{
  "query": {
    "project_ids": [502],
    "car_model_id": "lavida_2021_15l"
  },
  "items": [
    {
      "quotation": {}
    }
  ]
}
```

说明：
- 行情参考价和附近商户实时报价是两种不同能力
- 结果结构可以相似，但语义要明确区分

---

## 六、重构建议

按落地顺序，建议这么推进：

1. 先统一命名
- 把旧命名全部替换成统一业务命名

2. 再重构返回结构
- 先做商户
- 再做项目
- 最后做报价

3. 最后再写 API 文档
- 文档不是照着后端 DTO 抄
- 文档应该围绕“这个 API 解决什么任务、返回什么业务结果”来写

---

## 七、当前最重要的结论

最核心的一句话是：

- agent-facing API 返回的应该是“任务结果结构”，不是“后端原始数据结构”

这件事如果定下来，后面很多复杂度都会自然下降。
